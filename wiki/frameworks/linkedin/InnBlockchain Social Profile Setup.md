# Social Profile Setup

> [!INFO] One-time setup workflows for institutional social presence · cross-phase framework
> Covers LinkedIn Company Page, X (Twitter) personal handle, GitHub presence, and Solodit profile. Each is set up once when the phase activates the surface; ongoing-ops cadences live with the phase playbook (cadences shift per phase load). Re-use this framework verbatim when Phase 2+ activates a new entity (e.g., new corporate handle, new methodology surface).

---

## LinkedIn Company Page

*(This is `linkedin.com/company/<handle>` — the LinkedIn-side Page the founder posts company content from. Separate from the marketing website, which is Dev-owned.)*

### Initial setup (one-time)

- [ ] **(Vignesh)** LinkedIn Company Page exists (claimed/created on LinkedIn) with Vignesh as Page admin
- [ ] **(Vignesh)** CTA button on LinkedIn Page: "Contact us" → discovery call link (NOT a 404 destination)
- [ ] **(Vignesh)** Vignesh linked as employee on his personal profile (cross-association boosts organic reach when he posts)
- [ ] **(Vignesh)** Showcase Pages deferred until later phase (per LinkedIn Marketing Strategy Cross-Channel Destination Dependency warning — Showcase Pages reference LPs that may not exist yet)

### Completeness audit (one-time at setup + quarterly)

> [!INFO] Setup makes the Page live; completeness makes it credibility-grade
> Company Page is the institutional surface — vendor risk reviewers, recruiters, partners, and compliance officers check it as part of vendor diligence. A live-but-empty Page reads as a shell company.

- [ ] **(Vignesh authors · MPs maintain)** About section: ~2,000-char company description with regime positioning — must match Vignesh profile About claim verification (no contradictions on years · audit count · segment scope)
- [ ] **(Vignesh)** Industry: **Financial Services** (NOT "Software Development" — categorisation determines who LinkedIn shows the Page to)
- [ ] **(Vignesh)** Company size band selected
- [ ] **(Vignesh)** Headquarters location + founded year
- [ ] **(Vignesh)** Website link verified live (not 404 — re-check after any LP migration)
- [ ] **(Designer · MPs apply)** Banner image: current-phase brand template (refresh anchored to phase milestones)
- [ ] **(MPs)** Specialties field — match top 3 + 2 of Vignesh profile Skills

### Ongoing operations (post-setup, weekly)

> [!INFO] Page-specific content types, NOT personal-feed re-broadcasts
> Treat Page as a separate publishing destination with its own cadence (3×/wk) and content mix — case studies · milestones · service availability · industry commentary — not a duplicate of Vignesh's personal feed.

#### Posting cadence + mix

- [ ] **(MPs)** 2× repurposed Vignesh posts/wk — ≥48 hr lag from personal post (avoid double-feed cannibalisation)
- [ ] **(Vignesh authors · MPs schedule)** 1× company-specific post/wk — case study OR milestone OR service post OR industry commentary
- [ ] **(MPs)** Alternate days with Vignesh personal — never both same day on same topic
- [ ] **(MPs)** 3–5 hashtags per post (same algorithm rules as personal feed)

#### Page-specific content types

- [ ] **(Vignesh + delivery team)** Case study posts — fire when case study gates clear (anonymised + screenshot-cleared)
- [ ] **(Vignesh)** Milestone posts — first audit shipped · first platform live · phase milestones
- [ ] **(Vignesh + MPs)** Service-availability posts — "vendor risk pack available on request" · gated download CTAs · "sample audit report on request"
- [ ] **(Vignesh + MPs · institutional voice not first-person)** Industry commentary — re-share Vignesh personal hot-take from Page in slightly more formal voice
- [ ] *(DEFER until team grows past founder + ~5)* Employee spotlight posts

#### Analytics tracking (MPs report monthly · Vignesh reviews)

- [ ] **(MPs)** Page follower growth — target +25–50/mo
- [ ] **(MPs)** Page-view sources (referrer = personal feed · search · external) — Vignesh employee cross-association should drive ~40% of Page views
- [ ] **(MPs)** Post engagement rate per content type — feed into monthly review pattern-match
- [ ] **(MPs)** CTA button click rate — if "Contact us" → discovery call gets <5 clicks/mo, escalate (Page underperforming · revisit content cadence)
- [ ] **(MPs · Vignesh reviews)** Compare Page engagement vs Vignesh personal feed — Page typically ~10–20% of personal feed reach · <5% = problem signal

#### Showcase Pages revisit criteria (deferred per setup; revisit when ALL true)

- [ ] At least 1 next-phase sub-page LP shipped
- [ ] Page follower count >500 (Showcase Pages divide audience — only worth it past 500)
- [ ] 2+ distinct ICP segments active (different content speaks to different audiences)
- [ ] Current phase close cleared

> [!WARNING] Page hygiene hard rules
> - **(MPs)** Never post Company Page same day as personal feed on same topic (cannibalises reach)
> - **(MPs · Vignesh enforces)** Never use legacy-company messaging or off-positioning framing on this Page — pure current-company positioning
> - **(Vignesh quarterly)** Page admin access list audited — only Vignesh + 1 MP have posting access · never more
> - **(MPs)** Cross-association integrity: Vignesh's personal Experience entry must always link to the Company Page (verify after any profile edit · breaks if Company Page URL changes)

## X (Twitter) personal handle

### Initial setup (one-time)

> [!INFO] Why X matters
> Track A derivative chain produces an X thread per Tier 1/2/3 article. Algorithm boost requires baseline profile authority + warmup similar to LinkedIn. Categorisation problem applies — existing handle (if any) likely needs re-categorising to the current positioning.

- [ ] **(Vignesh)** X account claim — handle ideally matches LinkedIn · existing handle audit if reusing
- [ ] **(Vignesh decision)** Premium subscription decision — required for long-form thread tweets (>280 char/tweet) · algorithm boost · video uploads >2 min · Spaces hosting. **Recommendation: Premium from day 1** given thread volume.
- [ ] **(Vignesh)** Bio in regulatory + positioning framing — match LinkedIn About first sentence (correct regime tags · never wrong-regime framing for the current phase ICP)
- [ ] **(Vignesh + MPs · Designer template)** Header image: brand template (match LinkedIn profile banner for cross-channel consistency)
- [ ] **(Vignesh)** Location: EU city matching LinkedIn (NCA jurisdiction signal)
- [ ] **(Vignesh)** Website link: corporate domain (verified live, not 404)
- [ ] **(Vignesh)** Pinned tweet: 1 high-signal thread (preview of highest-leverage Tier 1 article OR Sample Audit + Risk Checklist call-out)

### Completeness + recategorisation (parallel to LinkedIn recategorisation)

> [!IMPORTANT] X categorisation problem
> Same as LinkedIn — existing X signal (if any) is likely off-positioning. The algorithm needs to re-categorise. Run X recategorisation parallel to LinkedIn Profile Recategorisation Accelerator.

- [ ] **(Vignesh + MPs)** Following list audit — unfollow off-topic accounts · follow 50–100 carefully chosen accounts in the current-phase ICP space (not bulk)
- [ ] **(Vignesh)** X Communities membership — 3–5 active communities matching current-phase ICP
- [ ] **(MPs · Vignesh review)** Cross-association: X handle linked from LinkedIn Contact Info + LP About page + Featured assets footer
- [ ] **(SP · Vignesh contributes ≤15 min/day)** First-2-weeks engagement: 5–10 reply-comments/day on current-phase ICP X accounts

### Ongoing operations (post-setup, weekly)

> [!INFO] X-specific cadence
> Less than LinkedIn (X audience less institutional). Native scheduler per [[InnBlockchain - Per-Platform Posting Spec]].

#### Posting cadence + mix

- [ ] **(MPs schedule · Vignesh authors · CW outlines thread)** 1× X thread/wk per Track A article (Tier 1 full · Tier 2 lite · Tier 3 amplify-only)
- [ ] **(Vignesh authors · MPs schedule)** 2–4× standalone X posts/wk (commentary · poll · hot-take on current-phase ICP topics)
- [ ] **(SP)** 5–10 strategic replies/day on current-phase ICP X thought-leader accounts
- [ ] **(Vignesh decision · MPs coordinate)** 1× X Space per quarter — IF Premium acquired AND co-host fits. Defer unless inbound Spaces invitation lands first.

#### Analytics tracking (MPs report monthly · Vignesh reviews)

- [ ] **(MPs)** Follower growth — target +50–100/mo
- [ ] **(MPs)** Thread completion rate per Tier 1 X thread — drop-off pattern feeds next thread structure
- [ ] **(MPs)** Click-through to LP from X bio link + thread last-tweet link — target ≥2% on Tier 1
- [ ] **(MPs · Vignesh decision at monthly)** Compare X engagement vs LinkedIn engagement — if X consistently <20% of LinkedIn reach for 4+ consecutive weeks, reduce X cadence to amplify-only

> [!WARNING] X hygiene hard rules
> - **(Vignesh)** Never post off-positioning content from this handle post-recategorisation (never confuse the signal)
> - **(Vignesh + MPs)** Never use wrong-regime framing for the current phase (#1 disqualifier)
> - **(Vignesh)** Never engage in industry drama / shitposting / vendor disputes — institutional positioning depends on signal-only behavior
> - **(MPs)** Never schedule X thread same day as LinkedIn long-form post on same article (cross-platform cannibalisation; ≥24 hr stagger per [[InnBlockchain - Cross-Channel Cannibalisation Matrix]])

> [!INFO] Corporate X handle decision
> X does not have a structural Company Page concept like LinkedIn or Facebook (every account is just an account). Phase 1 default: corporate X handle = NOT created. Revisit only if (a) X drives material LP traffic OR (b) ecosystem-engagement use case surfaces.

## GitHub + Solodit — methodology credibility surfaces

> [!INFO] Different shape than LinkedIn / X · owned by the Crypto Native pod
> GitHub + Solodit are credibility surfaces for **technical buyers** (Felix CTO vetting + Sasha-grade audit credibility) — NOT publish channels with weekly cadence. One-time setup + per-contribution cadence (per Tier 1 article that has a code or methodology angle). **These sit in the Crypto Native funnel, so the CN-pod marketer owns + coordinates the surface** (keeps it live · tracks credibility signal · tags repos per article) — but the **technical content is executed by the senior engineer + founder** (owner ≠ executor). *Phase 1: Britto owns/coordinates.*

### GitHub presence (CN-pod marketer owns/coordinates · Vignesh org-admin · CW + senior engineer execute)

- [ ] **(Vignesh)** Org claim — `github.com/<orgname>` · 2FA required on all admins
- [ ] **(CW drafts · Vignesh + senior engineer review)** Org README: current-phase positioning · methodology overview · link to primary LP
- [ ] **(senior engineer · Vignesh review)** 1–2 reference repos: methodology examples · standards implementation patterns · reference implementations
- [ ] **(CN-pod marketer coordinates · senior engineer executes)** Per Tier 1 Track A article that mentions code: tag the repo as canonical source · link the article from repo README
- [ ] **(Vignesh quarterly)** Org settings audit: members · admin access · 2FA enforcement · SECURITY.md file present + accurate

### Solodit profile (CN-pod marketer owns/coordinates · Vignesh + senior engineer execute)

- [ ] **(Vignesh + senior engineer)** Profile claim — auditor handle matches LinkedIn + GitHub
- [ ] **(senior engineer)** Methodology profile content — vulnerability classes you've reported · severity framework (matches Sample Audit Report PDF framework) · disclosure track record
- [ ] **(senior engineer · per published audit finding)** Submit findings from public disclosure to Solodit — minimum: 2–3 historical findings before launch · ongoing: every new disclosure
- [ ] **(Vignesh monthly)** Review Solodit profile views + finding citations · pattern-match against Sasha-grade discovery call mentions
