# ERC-1400: Security Token Standard

ERC-1400 is an Ethereum token standard for **security tokens** — digital tokens representing regulated real-world assets (stocks, bonds, real estate, fund shares). It extends ERC-20 with a mandatory compliance and regulatory layer directly inside the smart contract.

> ERC-1400 = ERC-20 + Legal Compliance + Transfer Controls + Partitioned Ownership

---

## Why Not ERC-20?

| Feature | ERC-20 | ERC-1400 |
|---------|--------|----------|
| Basic transfer | Yes | Yes |
| KYC/AML enforcement | No | Yes |
| Transfer restrictions | No | Yes |
| Forced transfers (court orders) | No | Yes |
| Document attachment (prospectus) | No | Yes |
| Partitioned tokens (tranches) | No | Yes |
| On-chain identity verification | No | Yes |

ERC-20 has no built-in way to restrict who can hold or transfer a token — not suitable for regulated securities.

---

## Architecture: A Family of Sub-Standards

```
ERC-1400 (Umbrella Standard)
│
├── ERC-1594  →  Core Security Token (issuance, redemption)
├── ERC-1410  →  Partially Fungible Tokens / Partitions (tranches)
├── ERC-1643  →  Document Management (attach legal docs on-chain)
└── ERC-1644  →  Controller Operations (forced transfers for regulators)
```

---

## Key Features

### Partitions (Tranches) — via ERC-1410

A single token can carry **multiple share classes** within one contract (e.g., Class A voting vs. Class B non-voting, or senior vs. junior tranche in real estate).

Real-world example: a tokenized property with:
- `SENIOR_TRANCHE` — priority income, lower risk (institutional investors)
- `JUNIOR_TRANCHE` — higher return, higher risk (retail investors)

### Transfer Restrictions — `canTransfer()` Compliance Hook

Before every token transfer, the contract checks:
- Is the recipient KYC verified?
- Is the investor accredited (for Reg D securities)?
- Is the lock-up period still active?

**ESC (Ethereum Status Codes):**

| Code | Meaning |
|------|---------|
| `0x51` | Transfer Allowed |
| `0x50` | Transfer Denied |
| `0x55` | Funds Locked |
| `0x56` | Transfer Verified Off-chain |

### Issuance and Redemption — via ERC-1594

- **Issue:** Mint tokens to KYC-verified investor wallets, controlled by the issuer role.
- **Redeem:** Investor burns tokens; off-chain process wires fiat to their bank account.

### Document Management — via ERC-1643

Legal prospectuses, SPV agreements, and regulatory disclosures are anchored on-chain:

```
Document PDF → IPFS upload → SHA256 hash → stored in smart contract
```

Regulators can verify on-chain that the document hash matches what was disclosed. Tamper-proof.

### Controller Operations (Forced Transfers) — via ERC-1644

A regulator-appointed controller can forcibly move tokens in cases of court orders, fraud recovery, or AML seizure. Controversial but legally required for securities law compliance.

## Token Lifecycle

```
1. Deploy ERC-1400 Contract
2. Set Up KYC Registry + Whitelisted Investors
3. Attach Legal Documents (Prospectus, SPV Agreement)
4. Issue Tokens to Investors
5. Investors Trade (canTransfer() checks every move)
6. Dividends / Yield Distributed via smart contract
7. Investor Redeems Tokens → Receives Fiat
```

## ERC-1400 vs. ERC-3643

| Feature | ERC-1400 | ERC-3643 |
|---------|----------|----------|
| Origin | Open-source community | Tokeny Solutions |
| Identity standard | Custom KYC mapping | ONCHAINID (ERC-734/735) — standardized |
| Adoption | Broader in open-source | Stronger in institutional Europe |
| Modularity | High (sub-standards) | High (modular compliance modules) |
| Preferred for | General RWA dev | GDPR-regulated, European markets |

See [[ERC-3643]] for the competing standard.

Building an ERC-1400-based RWA platform:

| Module | Work Involved |
|--------|--------------|
| Security Token Contract | Custom ERC-1400 contracts with investor restrictions |
| KYC Registry | On-chain whitelist tied to identity provider (Onfido, Sumsub) |
