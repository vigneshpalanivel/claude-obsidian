# ERC-3643: T-REX Security Token Standard

**ERC-3643**, also called **T-REX (Token for Regulated EXchanges)**, is an Ethereum token standard for compliant security tokens. Developed by **Tokeny Solutions** and ratified as an EIP, it is the more structured, identity-centric alternative to [[ERC-1400]].

Key differentiator: ERC-3643 integrates with **ONCHAINID** (ERC-734/ERC-735) to give every investor a standardized, self-sovereign on-chain identity — rather than a custom per-issuer KYC mapping.

> ERC-3643 = Security Token + On-Chain Digital Identity (ONCHAINID) + Modular Compliance Engine

Especially strong for **European securities regulation (MiFID II, GDPR)** and widely adopted by institutional issuers.

---

## ERC-1400 vs. ERC-3643

| Feature | ERC-1400 | ERC-3643 |
|---------|----------|----------|
| Identity standard | Custom KYC mapping | ONCHAINID (ERC-734/735) — standardized |
| Compliance engine | Manual rules in contract | Modular, pluggable compliance modules |
| Regulatory focus | General/global | Strong fit for EU (MiFID II, GDPR) |
| Developer ecosystem | Open-source community | Tokeny + institutional partners |
| Adoption | Widely used in open-source | Preferred in European institutional markets |

## Architecture: 4 Components

```
ERC-3643 ECOSYSTEM
│
├── Token Contract (T-REX)        — ERC-20 compatible token with compliance hooks
├── Identity Registry (ONCHAINID) — every investor has a verified on-chain identity
├── Compliance Module Engine      — modular, pluggable rules engine
└── Trusted Issuers Registry      — defines which KYC providers are trusted
```

Each transfer calls the compliance module, which reads from the identity registry, which checks claims signed by trusted issuers. Any failed check reverts the transfer with a reason code.

---

## ONCHAINID: The Key Differentiator

Every investor gets a **decentralized on-chain identity** — a smart contract wallet storing verified claims:

```
Investor's ONCHAINID
├── Claim: "KYC Verified"          — issued by Onfido (Trusted Issuer)
├── Claim: "Accredited Investor"   — issued by Compliance Firm
├── Claim: "Country: France"       — issued by KYC Provider
└── Claim: "Age > 18"              — issued by Identity Verifier
```

Each claim is:
- Cryptographically signed by the trusted issuer
- Stored on-chain in the investor's ONCHAINID contract
- Verifiable without exposing raw personal data (privacy-preserving, GDPR-compatible)

## Token Transfer Flow

```
Investor A wants to transfer 100 tokens to Investor B
        │
        ▼
1. Token contract calls canTransfer(A, B, 100)
        │
        ▼
2. Compliance Module checks:
   ├── Does B have a valid ONCHAINID?
   ├── Does B's ONCHAINID have required claims? (KYC, country, accreditation)
   ├── Are the claims from a Trusted Issuer?
   ├── Are lock-up or holding restrictions active?
   └── Do any jurisdiction rules block this transfer?
        │
        ├── All checks pass → Transfer executes
        └── Any check fails → Transfer reverted with reason code
```

---

## Compliance Modules

Pluggable rules swappable without redeploying the token contract:

| Module | Rule |
|--------|------|
| `MaxInvestorsModule` | Limits total number of token holders |
| `CountryAllowListModule` | Whitelist of allowed countries |
| `CountryRestrictModule` | Blacklist of restricted countries |
| `TimeLocksModule` | Lock-up periods after issuance |
| `TransferFeesModule` | Automatic fee deduction on transfer |
| `MaxOwnershipModule` | Limits one investor's max holding percentage |

## Regulatory Compliance Built In

| Regulation | How ERC-3643 Handles It |
|-----------|------------------------|
| EU MiFID II | Identity registry + investor classification claims |
| GDPR | Personal data stored off-chain; only hashed claims on-chain |
| US Reg D (506c) | Accredited investor claims verified by trusted issuers |
| US Reg S | Country restriction modules block US persons |
| AML/KYC | All transfers require verified ONCHAINID with KYC claim |
| FATF Travel Rule | Investor identity traceable for large transfers |

## Full Issuance Lifecycle

```
1. Deploy ONCHAINID contracts for each investor
2. Trusted KYC providers verify investors, issue on-chain claims
3. Asset issuer deploys T-REX Token + registers:
   ├── Identity Registry
   ├── Trusted Issuers Registry
   ├── Claim Topics Registry
   └── Compliance Contract
4. Tokens issued (minted) to verified investor wallets
5. Investors trade — all transfers auto-checked by compliance
6. Dividends/yield distributed by smart contract
7. Investor redeems or exits — tokens burned, fiat returned
```

## When to Use ERC-3643 Over ERC-1400

Prefer ERC-3643 when:
- Clients are in European or Asian regulated markets
- Strong privacy compliance (GDPR) is required
- Multiple compliance rules need to be swapped without redeploying the token
- Institutional investors require standardized digital identity integration
